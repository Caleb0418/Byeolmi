# 5Whys 분석 보고서: 공동구매 주문 관리 서비스 기획 검증

식품 도매업체의 공동구매 요청 및 수기 주문 취합 문제를 해결하기 위해 시니어 스타트업 매니저의 관점에서 5Whys 분석을 통해 근본 원인을 도출하고 제품 정의를 검증합니다.

---

## 1. 5Whys 분석 프로세스

### **Q1. 왜 사장님은 하루 종일 카톡을 보며 주문을 직접 엑셀/수기로 취합하는가?**
* **A1.** 업자들이 표준화된 양식 없이 각자 편한 방식(텍스트, 사진, 음성 등)으로 카톡 메시지를 보내기 때문에, 사장님이 일일이 내용을 해석하고 정제해야 하기 때문입니다.

### **Q2. 왜 업자들은 표준화된 시스템(주문 앱 등)을 쓰지 않고 굳이 개인 카톡으로 보낼까?**
* **A2.** 기존 카톡 주문이 가장 빠르고 익숙하여 자신들의 행동 흐름을 깨기 싫어하고, 새로운 주문 앱을 다운받아 로그인하고 품목을 찾아 주문하는 과정 자체가 '귀찮고 불필요한 비용'으로 느껴지기 때문입니다.

### **Q3. 왜 사장님은 개별 취합뿐만 아니라 배송/마감 전 과정을 머릿속이나 수기로만 기억하며 처리할까?**
* **A3.** 단순 취합 외에도 **'누적 공동구매 수량에 따른 품목별 단가 변동'**이 실시간으로 발생하고, 마감 시점에 각 업자에게 최종 확정 단가와 입금액을 수동으로 안내 및 매칭해야 하기 때문입니다. 즉, 단순 주문 접수 시스템만으로는 유동적인 정산 예외 처리가 어렵습니다.

### **Q4. 왜 최종 단가 결정과 정산 안내가 이토록 수동적이고 불투명하게 진행되는가?**
* **A4.** 업자들이 주문할 당시에는 오늘 최종 단가가 얼마가 될지 알 수 없고, 마감이 끝난 후에야 전체 주문 수량이 확정되어 최종 단가가 계산되기 때문입니다. 사장님은 정산 금액을 투명하게 공유할 실시간 공유 채널이나 도구가 없어 매번 개별 카톡으로 통보하고 있습니다.

### **Q5. 왜 실시간 단가 변동과 정산금액이 투명하게 공유되지 않아 발생하는 비효율이 고착화되었는가? (근본 원인)**
* **A5.** **주문 접수(업자 관점)는 극도로 단순해야 하지만, 공동구매 진행 상황(달성률 및 현재 예상 단가)과 마감 정산(사장님-업자 간)을 실시간으로 싱크해줄 수 있는 '통합 데이터 파이프라인 및 경량화된 인터페이스'가 없었기 때문입니다.**

## 2. 스타트업 관점의 비즈니스 및 제품 설계 제안

업자들의 카톡 사용 행동 패턴을 100% 유지하면서 사장님의 업무를 자동화하기 위한 최적의 해결책(Solution Blueprint)을 제안합니다.

### 💡 솔루션: **카카오톡 채널 챗봇(업자용) + 반응형 관리자 웹 대시보드(사장님용)**

```mermaid
graph TD
    A[업자 20~30명] -->|1. 카카오 채널 챗봇 주문| B(카카오톡 싱크/챗봇)
    B -->|2. 주문 API 전송| C[백엔드 데이터베이스]
    C -->|3. 실시간 취합 및 예상 단가 계산| D[사장님용 웹 대시보드]
    D -->|4. 원클릭 마감 처리| C
    C -->|5. 마감 알림톡 발송 (최종 단가 및 입금 계좌)| A
```

### 1) 업자용 인터페이스 (행동 변화 최소화)
* **카카오 채널 챗봇 도입**: 별도 앱 설치 불필요. 카카오톡 내에서 버튼 클릭 몇 번으로 주문서 작성(예: 품목 선택 -> 수량 입력 -> 주문 완료).
* **예상 단가 실시간 확인**: 챗봇 하단 메뉴에 '오늘의 공구 현황'을 제공하여, 현재까지 누적 수량과 적용 예정 단가 구간을 업자가 직접 조회할 수 있게 함으로써 사장님에게 문의하는 카톡 빈도를 획기적으로 차단.

### 2) 사장님용 관리자 웹 (수기 업무 제로화)
* **실시간 취합판**: 카톡 채널을 통해 들어온 주문이 자동으로 품목별/업자별로 정렬되어 대시보드에 반영.
* **단가 자동 계산기**: 품목별 누적 수량에 따라 적용 단가가 실시간으로 자동 슬라이딩 적용되어 수식 계산 필요 없음.
* **원클릭 정산 & 마감**: 마감 버튼 클릭 시, 오늘 주문한 업자들에게 **최종 확정 수량, 최종 단가, 총 입금액, 입금 계좌**가 카카오 알림톡(템플릿 메시지)으로 자동 발송.

## 3. 제약사항 극복 및 MVP 실행 전략

* **기술적 제약 극복**: 사장님 개인 카톡은 파싱이 불가능하므로, 사장님의 프로필 대신 **'업체 공식 카카오톡 채널'**을 개설하게 유도합니다. "이제 채널로 주문하셔야 실시간 공구 할인율을 보고 더 저렴하게 가져가실 수 있습니다"라는 혜택 중심의 커뮤니케이션으로 업자들을 채널로 자연스럽게 온보딩합니다.
* **데이터 규모 제약**: 품목 수 20개 이하, 업자 20~30명 수준이므로 복잡하고 무거운 DB 설계 대신 빠르게 검증 가능한 **Supabase + Next.js** 혹은 경량 데이터베이스 기반으로 2주 안에 MVP를 빌드하여 실사용 피드백을 수집할 수 있습니다.
